# walkmap — architecture & development notes

> Audience: developers (human or LLM) extending this codebase. Everything here
> was learned the hard way on 2026-07-04/05; the **Gotchas & dead ends**
> section is the most valuable part — read it before "improving" anything.

## What this is

For each configured city, walkmap partitions the OSM walking network into
**catchments**: the set of places whose *walking-time-nearest* instance of an
amenity type (supermarket, pub, bus stop, …) is the same one. It is a network
Voronoi diagram where the metric is elevation-aware walking time on a
**directed** graph (uphill ≠ downhill), rendered as an interactive vector-tile
map with two modes (buildings coloured by catchment + time; catchment area
polygons) and a hover-traced shortest walking path.

Live: https://sztanko.github.io/walkmap/ · 24 cities as of 2026-07-05.

## Repo layout

```
config/cities.yaml          city registry: pbf_url, country, curated `types`, bbox, grid_m
config/feature_types.yaml   feature-type catalogue with per-country/city `variants`

web/                        static UI: index.html, app.js, style.css — NO build step
web/data/manifest.json      generated by `walkmap manifest`; committed (deployed with site)
scripts/release.sh          publish one city's tiles to its Pages data repo
scripts/rebuild.sh          run+publish one/many/all cities; --analyze; --no-publish
data/                       gitignored: pbf/ dem/ work/<city>/ out/<city>/ logs/ analyze/
.github/workflows/pages.yml deploys web/ to GitHub Pages on push to main
```

## Technology choices (and why)

| Choice | Why |
|---|---|
| **Rust** pipeline (single binary) | Greater London = ~2M-node graph, 1.16M buildings, parsed from a 121MB PBF in ~1 min. Python alternatives (pyrosm materializes GeoDataFrames; per-element pyosmium callbacks; GEOS unions) blow either 24GB RAM or the time budget. |
| `osmpbf` + rayon `par_bridge` over blobs | Parallel two-pass streaming PBF decode without materializing. **Must sort results by OSM id after reduce** — see Gotcha #5. |
| Custom CSR graph, `u32` decisecond weights | Static graph; petgraph's adjacency objects are pointless overhead at this scale. Only the REVERSED graph is built (see Flow step 5). |
| `kiddo` kd-tree (f64, equirectangular meters) | Nearest-graph-node snapping for features/buildings/grid cells. City-scale equirectangular projection error is negligible. |
| Copernicus DEM GLO-30 (AWS open bucket, whole 1° COG tiles) | Free, global, consistent quality (SRTM has voids; terrarium tiles have mosaic seams that alias into slope). Whole-tile download beats thousands of range reads; `tiff` crate decodes them fine. |
| Tobler's hiking function | Single smooth directional slope→speed formula: `v = 6·e^(−3.5·|slope+0.05|)` km/h; slope clamped ±0.35, zeroed on bridges/tunnels (DEM sees terrain, not the deck) and segments <3m. Steps get fixed 0.45/0.6 m/s up/down. |
| Multi-source Dijkstra on reversed edges | One pass per feature type seeds ALL sites at 0; label + distance + **next_hop** per node. Reversed arcs make the result "time from node TO its nearest site", as required. Seconds per type even for London. |
| 10m raster grid → polygonize (NOT exact per-node Voronoi union) | GEOS-style union of millions of sliver cells is hours per city. Grid labelling + boundary tracing is minutes, and NODATA cells (>250m from any node) naturally punch out rivers/parks. |
| tippecanoe (felt fork ≥2.17) → **PMTiles** | Writes `.pmtiles` directly; per-feature `{"tippecanoe":{"layer":…,"minzoom":…}}` JSON keys let one invocation build both layers with different zoom ranges. |
| **GitHub Pages per-city data repos** (`walkmap-data-<city>`) | GitHub **Releases serve no CORS headers** (verified — browser fetch impossible). github.io serves `access-control-allow-origin: *` + Range 206. Limits: 100MB/file hard (hence tile maxzoom z15), ~1GB/site soft (London serves fine at 1.35GB). |
| MapLibre GL JS + pmtiles.js from CDN, OpenFreeMap `positron` basemap | No build step, no API keys. Colours are baked into tiles at build time (attr `c`), so the client style is trivial. |

## Pipeline flow (`walkmap run <city>`, see `pipeline/src/main.rs::run_city`)

1. **Fetch** PBF via system `curl` (NOT ureq — see Gotcha #1) into `data/pbf/`, cached.
2. **Extract** (`osm.rs`): two passes. Pass 1 (ways): walkability filter
   (`walkable()` — highway whitelist, foot/access overrides, ferry exclusion,
   trunk-with-sidewalk exception), building ways, feature-type tag matching
   (`config::MatchRule`, resolved per city). Pass 2 (nodes): coords for needed
   ids within bbox + tagged feature nodes. Assembly: compact node ids
   (**every** way node is a graph node — no intersection contraction; density
   helps snapping/labelling), segments, building rings+centroids, features per
   type. Cached as `extract_<hash>.bin` where hash = fnv of the **resolved**
   feature config (`config::types_hash`).
3. **Prune** weak components <30 nodes (union-find), remap ids (`graph.rs`).
4. **Elevation** (`elevation.rs`): sample every node bilinearly. Cache
   `elevv2.bin` = `(node_fingerprint, Vec<f32>)` — see Gotcha #5.
5. **Reversed weighted CSR** (`graph.rs::build_rev_csr`): for segment (a,b),
   reversed arc b→a carries forward time t(a→b) and vice versa.
6. **Snap** (`snap.rs`): features ≤200m, buildings ≤300m to nearest node; the
   building's time later = node time + snap_dist/1.39 m/s.
7. **Grid** (`grid.rs`): per-cell nearest node + snap distance (u16 dm) at
   `grid_m` (10m default, 5m small cities). Cached `gridv3_<m>.bin` =
   `(node_fingerprint, Grid)`. NODATA where nothing within 250m.
8. **Per feature type**:
   a. **Group** near-duplicate features (`group.rs`): union-find, ≤100m apart
      AND similar normalized names (equal or prefix). One site (pid) per
      group; **every member's snapped node is seeded** → catchment is the
      union of both directional stops. `k` = member count → UI "(×2)".
   b. **Dijkstra** → per node: label (pid), dist (ds), next_hop.
   c. Cell labels = `label[grid.nearest[cell]]`; **absorb islands** smaller
      than 1200m² into the neighbour with the longest shared border
      (`polygonize::absorb_small_islands`) — bigger islands are genuine
      network effects (e.g. a levada path reaching a different bus stop).
   d. **Polygonize** (`polygonize.rs`): boundary edges grouped into SHARED
      polylines per unordered label pair (junction→junction); each polyline
      simplified (DP ε=0.9 cells) + Chaikin×2 ONCE so both neighbours reuse
      identical geometry (⇒ smoothing can never open gaps). Stitch directed
      polylines per label into rings (sharpest-right-turn at saddles),
      shells/holes by shoelace sign, holes nested by point-in-polygon.
      Also returns the **adjacency** (label pairs) for free.
   e. **Colour** (`output.rs::assign_colors`): greedy 6-colour graph
      colouring, highest degree first — adjacent catchments never share a
      colour (planar ⇒ 6 always enough). Colours are BAKED into tiles:
      partitions get palette hex; buildings get palette hue darkened by
      walking time (30-min ramp), attr `c`.
   f. **Write** GeoJSONSeq (partitions MultiPolygons: pid,name,t_max,c;
      buildings: pid,t,c) → **tippecanoe** `-Z7 -z15 --detect-shared-borders
      --coalesce-smallest-as-needed --drop-densest-as-needed` → one
      `<type>.pmtiles` with layers `partitions` (all zooms) and `buildings`
      (per-feature minzoom 13). z15 max keeps every archive <100MB.
   g. **Path raster** (`grid.rs::direction_field`): one byte/cell, 0=terminal,
      1–8=step direction. Built by painting each Dijkstra tree edge as an
      8-connected Bresenham line **with per-cell interpolated-time priority**
      (see Gotcha #8), then off-street cells point one step toward their
      nearest node's cell. Gzipped → `<type>.dirs.gz`.
   h. `<type>.sites.json`: `{"sites":[[pid,name,lng,lat,k],…]}` for search,
      popups, site dots.
9. **Meta/manifest**: `meta.json` (bbox + pathGrid geometry), stale outputs of
   de-configured types deleted; `walkmap manifest` writes
   `web/data/manifest.json` (cities+types+bbox+pathGrid + **`v` cache-buster**,
   see Gotcha #7).
10. **Publish** (`scripts/release.sh`): force-push tiles+sites+dirs+meta to
    `walkmap-data-<city>` repo, enable branch Pages (retried — creation races).

`walkmap analyze <city>` extracts ALL catalogue types and prints raw counts /
grouped sites / median nearest-neighbour spacing with verdicts — **run this
before configuring a city's `types`**; the user's rule is "community-defining
groups only" (e.g. Calheta has 1 supermarket → skip; London Greggs = 192 shops
→ keep, for fun).

## The web UI (`web/app.js`)

- URL hash `#city/type/mode/zoom/lat/lng`. Sources are torn down and rebuilt
  on city/type switch; tiles URL = `pmtiles://<base><type>.pmtiles?v=<manifest.v>`.
- Fill colour is just `["to-color", ["get","c"], "#999"]` — everything hard is
  precomputed in Rust.
- Buildings mode shows dash-dotted teal catchment borders (`areas-line`
  restyled in `applyMode`); areas mode shows white borders.
- Hover path: lazily fetch `<type>.dirs.gz`, gunzip via `DecompressionStream`
  (detect gzip magic bytes — github.io serves it raw), trace ≤60k steps with a
  visited-set, Chaikin once, render as a dense arrow chain (symbol layer,
  `symbol-placement: line`, ASCII `>` — fancy arrow glyphs are missing from
  the basemap's glyph pbfs). Leading 40ms throttle: tracing costs well under
  1ms; setData is the only real cost.

## Gotchas, dead ends, and the correct solutions

1. **ureq/rustls fails TLS on this network** (`invalid peer certificate:
   UnknownIssuer` — intercepting proxy). Dead end: bundling cert stores.
   Correct: shell out to system `curl` (`elevation::curl_download`) which uses
   the platform trust store.
2. **GitHub Releases have NO CORS** (checked GET and OPTIONS on
   `release-assets.githubusercontent.com` — no `access-control-allow-origin`
   at all). The original "PMTiles on Releases" plan is impossible in-browser.
   Correct: per-city GitHub Pages repos — github.io sends `ACAO: *` and
   supports Range (206). Consequences: 100MB/file hard cap → tile maxzoom 15
   (MapLibre overzooms losslessly, ~20cm precision); ~1GB/site soft cap is
   tolerated in practice (London: 1.35GB).
3. **`python3 -m http.server` ignores Range** → pmtiles breaks locally. Use
   `npx http-server -p 8000 -c-1` (`-c-1` matters: default caching is 1h and
   makes you debug ghosts).
4. **Per-layer zoom ranges in one tippecanoe run**: `-L` JSON doesn't do it;
   the working mechanism is the per-feature `"tippecanoe":{"layer":…,"minzoom":…}`
   object in each GeoJSON feature.
5. **THE BIG ONE — nondeterministic extraction scrambles index-aligned
   caches.** `osmpbf` parallel blob iteration yields ways in scheduler order,
   so compact node ids differ between extracts of identical data. The
   nearest-node grid and elevation array store node *indices*; validating them
   by length/range only, they silently married a NEW extract to OLD indices →
   partition labels effectively random (symptom: pub catchments in Orpington
   owned by the Euston Tap 23km away; confetti-mosaic areas; bloated tiles
   because nothing shares borders). Correct: (a) sort ways/features by OSM id
   after the parallel reduce (determinism), AND (b) store a **fingerprint of
   the node array** (`main.rs::nodes_fingerprint`) inside every index-aligned
   cache and validate on load. Do BOTH — determinism alone breaks again the
   day the walkable-filter changes.
6. **`cargo test --release` does not necessarily refresh
   `target/release/walkmap`.** A background batch picked up a stale binary and
   re-baked scrambled tiles once more. Always `cargo build --release` before
   invoking the binary from scripts (rebuild.sh does).
7. **Rewriting a .pmtiles under an open browser session serves garbage**: the
   pmtiles client caches the archive's internal directory by URL; new file +
   old offsets = valid-looking but wrong tiles. Correct: manifest carries
   `v` (unix time at generation), appended as `?v=` to every data URL; always
   regenerate + deploy the manifest when republishing tiles.
8. **Path direction raster — three dead ends before the right answer**:
   - *Per-cell time descent* (point at the 8-neighbour with smaller
     `t = node_time + snap/1.39`): dead-ends at every road node — all its
     neighbouring cells snap back to the same node with larger snap distance.
     Symptom: "trace stops at the nearest road".
   - *Head-to-your-node, then next_hop*: the cell you step onto after leaving
     a node's cell targets its own nearest node — straight back. 3–5 point
     traces.
   - *Painting Dijkstra tree edges with per-EDGE priority* (slowest first,
     fastest overwrites): traces reached the site but could hop onto a
     *slower* branch where lines cross and orbit blocks (visible loops).
   - **Correct**: paint tree edges as Bresenham lines where each cell accepts
     a direction only if this edge's **interpolated time at that cell** beats
     the best seen (`grid.rs::direction_field`). Branch switches then strictly
     decrease remaining time ⇒ loop-free, terminates at the site. Client
     keeps a visited-set as belt-and-braces.
9. **Path raster resolution must equal the area grid's.** At 2× coarser (a
   size optimization), hover cells snapped to different nodes than the area
   raster → traces visually started/crossed into foreign catchments, which is
   topologically impossible for true shortest paths (optimal substructure).
   Same `grid_m` for both ⇒ consistent; London dirs.gz ≈ 3.4MB gz / 31MB RAM,
   acceptable lazily-loaded.
10. **Smoothing polygon boundaries per-ring opens gaps.** Douglas-Peucker or
    Chaikin applied independently to each partition's ring makes shared
    borders diverge. Correct: extract each shared boundary polyline once
    (grouped by label pair), smooth it once, let both neighbours reuse it.
    Related: raw (unsimplified) shared edges also let tippecanoe's
    `--detect-shared-borders` work.
11. **Small catchment islands are usually snapping noise, but not always.**
    Absorb components <1200m²; keep the rest — they're real (network
    shortcuts: stairs, levadas, footbridges).
12. **Interchange stations fall through rail/metro splits**: OSM tags e.g.
    Finsbury Park as one node with `station=subway`, so a "rail (NOT subway)"
    group misses it entirely. The combined `station` group
    (railway=station|halt) is the robust default; per the user, splitting
    rail from underground is also *conceptually* wrong for residents.
13. **MapLibre expression traps**: there is no `hsl()` function and computed
    string→colour is fragile — bake final colours into tile attributes
    instead. Line-placed symbol glyphs must exist in the basemap's font pbfs
    (use ASCII `>`; `▶`/`→` may be tofu). `map.on("error")` — NOT the console —
    is where source/style errors surface.
14. **GitHub Pages branch builds of big repos wedge** (status "building",
    duration 0, for hours) and Pages-enable racing repo creation 404s.
    Correct: retry enable (release.sh does), and re-queue wedged builds with
    `gh api -X POST repos/<o>/<r>/pages/builds`. Deploys of the main site
    also intermittently fail with "try again later" — re-run the workflow.
15. **Geofabrik quirks**: some region URLs return HTTP 200 with an HTML error
    page (check magic bytes, not status); `portugal-latest` 302-redirects to
    a dated file (curl -L). Portugal has no sub-region extracts — Funchal,
    Calheta, Porto, Lisbon all share `portugal-latest.osm.pbf` (bbox-filtered
    at parse time).
16. **Grouping stops by name**: only merge features ≤100m apart with equal or
    prefix-normalized names; never merge unnamed. London: 19,203 bus stops →
    12,887 sites (6,146 directional pairs merged).

## Performance (M4 Pro, 24GB)

| City | graph nodes | buildings | full run (15–17 types) |
|---|---|---|---|
| Funchal (5m grid) | 100k | 46k | ~25s |
| Calheta | 40k | 20k | ~12s |
| London (10m grid) | 1.92M | 1.16M | ~15 min (~55s/type, 90% tippecanoe) |

Dijkstra: 1–3s per type on London. Polygonize+smooth: ~3s. Grid build: ~60s.

## Common tasks

- **Add a city**: append to `cities.yaml` (id, name, pbf_url, country, bbox,
  grid_m), run `scripts/rebuild.sh --analyze <id>`, choose `types` from the
  verdicts (≥~12 sites and community-defining; no rail/metro split; no
  combined `transit`; pubs UK-only), then `scripts/rebuild.sh <id>`, commit
  `web/data/manifest.json`, push.
- **Change a feature type's rules**: edit `feature_types.yaml` (variants keyed
  by country or city id override `match`); the extract cache re-keys itself
  automatically via the config hash.
- **Rebuild everything**: `scripts/rebuild.sh all` (hours; ~6GB of PBFs).
